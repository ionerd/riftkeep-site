RiftKeep

A free companion app for Riftbound — the League of Legends Trading Card Game: collection tracking, deck building with rules validation, and card loans between friends.

iOS & Android (React Native / Expo) Free — no monetization Unofficial fan project

1. Overview

RiftKeep is a mobile companion app for players of Riftbound, Riot Games' physical trading card game. It solves three everyday problems for players and playgroups:

  • Knowing what you own — tracking a physical collection per printing and per finish (normal / foil), including alternate-art showcase and secret variants.
  • Building legal decks — a deck builder that validates decks against the official construction rules and tells the player exactly which cards they still need to acquire.
  • Lending cards safely — a structured loan system between friends, so borrowed cards are never forgotten and collection counts stay accurate while copies are out on loan.

The app is a personal, non-commercial project. It contains no advertising, no in-app purchases, and no card sales or marketplace functionality of any kind.

2. Main Features

2.1 Card Catalog Browser

A browsable reference of the complete Riftbound card catalog (~1,100 cards), which is the foundation every other feature builds on.

  • Search & filters: instant name search plus one-tap filter chips by set, domain (Fury, Calm, Mind, Body, Chaos, Order), and card type.
  • Card detail view: card art, full name (name + subtitle), set and collector number, domains, type, and rarity.
  • Variant awareness: showcase alternate arts and secret overnumbered printings are modeled as distinct printings linked to their canonical card, so collectors can track each printing separately.
  • Legality & errata: the catalog carries format-scoped ban data with effective dates (from Riot's official Rules Hub) and dated, cumulative errata, both surfaced on the card detail screen.

2.2 Collection Tracking

Users record how many copies of each printing they physically own.

  • Per printing + finish: a copy is identified by card printing and finish (normal or foil), matching how collectors actually sort binders.
  • Quantity steppers: fast increment/decrement entry designed for cataloguing a binder in one sitting.
  • Owned vs. missing math: playset completion is aggregated by full card name (as the game's copy limits are), while quantities stay per printing.
  • Owned vs. available: the app distinguishes copies owned (physically yours) from copies available (owned minus copies currently lent out), and deck math uses the latter.

2.3 Deck Builder with Rules Validation

A full deck construction tool for Riftbound's deck structure: one Legend, a designated Chosen Champion, a Main Deck, a Rune Deck, and three Battlefields.

  • Zone-aware editing: cards are added to the correct zone (main / rune / battlefield), and the user picks the specific printing they want for its art.
  • Legality validation: decks are checked against the official construction rules — zone sizes, copy limits by full name, the Signature card limit, Domain Identity (the deck's colors must match its Legend), the Chosen Champion's tag matching the Legend, and current card bans with effective dates.
  • Deck checklist: for any deck, the app shows which required copies the user already has available and which are missing — the bridge between deck building and collection tracking.

2.4 Card Loans Between Friends

A structured, bilateral loan workflow so playgroups can share cards without losing track of them.

  • Loan lifecycle: proposed → accepted → active → returned, with declined / cancelled / expired (proposals lapse after 14 days) and a lender-side force-close for loans resolved outside the app.
  • Loan items & partial returns: each loan lists specific printings, finishes, and quantities; the lender confirms returns item by item, and the loan closes automatically when everything is back.
  • Action inbox: the app derives, per loan, the single action currently waiting on the user (answer a proposal, confirm receipt, confirm a return, or notice an expiring proposal) and badges the Loans tab with the count.
  • Accurate counts: lending never changes ownership — lent copies stay owned by the lender but stop being available, and are never counted in the borrower's collection.

2.5 Friends & Invites

  • Invite codes / QR: friendships are established via single-use, short-lived invite codes shown as a QR code — no search, no acceptance step, no discovery of strangers.
  • Friend collections: friends can browse each other's collections to see what could be borrowed before proposing a loan.
  • Minimal profiles: a user's public identity is a display name and nothing else.

2.6 Wishlist

Each collection entry carries a wanted quantity alongside the owned quantity; a dedicated wishlist screen aggregates everything the user is looking for — useful at trade tables and when friends check what to lend or gift.

3. How RiftKeep Uses the Riot Games API

RiftKeep needs exactly one thing from Riot: the Riftbound card catalog (card names, sets, collector numbers, domains, types, rarities, and card art references). No player data, no match data, and no account data are requested from any Riot API.

3.1 Architecture: server-side sync only

The mobile app never calls the Riot API directly. The catalog is synced server-side into the app's own read-only database (Supabase / PostgreSQL), and clients read only from that database:

AspectDesign
API consumerA single server-side sync job (Supabase Edge Function). Client apps have no Riot API access path.
API key storageServer-side environment secret only. The key is never embedded in the app binary, never shipped to devices, and never committed to source control.
Endpoint usageThe Riftbound Content API bulk catalog endpoint, fetched in full per sync run.
Request volumeThe catalog is near-static, so syncs are scheduled (on the order of one run per day) plus occasional manual runs on set releases or errata updates. Expected load: a handful of requests per day, independent of user count.
CachingThe synced catalog is versioned (version / lastUpdated); clients cache it for 24 hours and refetch only when the version changes.

3.2 Data handling

  • Catalog data is stored as a read-only reference table; user data (collections, decks, loans) references it by foreign key but never modifies it.
  • Card data is displayed for reference purposes within the app only — it is not re-exposed through any public API, feed, or export.
  • Ban list and errata data are kept in sync with Riot's official Rules Hub publications.

4. User Data & Privacy

  • Accounts exist only to persist a user's own collection, decks, and loans across devices (email-based authentication via Supabase Auth).
  • The only public profile field is a display name. No real names, avatars, locations, or Riot account links are collected.
  • All user data is protected by row-level security: users can read and write only their own data, plus the collections of confirmed friends (read-only) and loans they are a party to.
  • No analytics or tracking SDKs, no advertising identifiers, no data sharing with third parties.

5. Technical Summary

PlatformiOS and Android, built with React Native / Expo (Expo Router, React Query, NativeWind).
BackendSupabase: PostgreSQL with row-level security, Auth, and Edge Functions (the Riot API sync lives here).
Offline posturePersisted client-side cache so the catalog and the user's own data remain browsable offline; writes are gated on connectivity.
DistributionFree download; no ads, no in-app purchases, no marketplace or card-sale functionality.
