RiftKeep

A free companion app for Riftbound — the League of Legends Trading Card Game. Track a physical collection, build tournament-legal decks, and lend cards between friends.

What it solves

Three everyday problems for a Riftbound playgroup.

RiftKeep is a personal, non-commercial project. No advertising, no in-app purchases, and no card sales or marketplace of any kind — just the tools a play group actually needs around the physical game.

Know what you own

Your collection, per printing

Track every copy by printing and finish — normal or foil — including showcase alternate arts and secret overnumbered variants.

Build legal decks

Validated against the rules

A deck builder that checks a build against Riftbound's official construction rules and shows exactly which cards you still need.

Lend safely

Loans that stay accounted for

A structured loan workflow so borrowed cards are never forgotten and collection counts stay accurate while copies are out.

A walk through the app

Every screen is built on the Riftbound card catalog.

Screens below are captured on device. The interface ships in Italian — the app's launch locale — while card data stays in its original English.

Catalog & sets

Browse the whole catalog, or one set at a time.

The catalog is a browsable reference of the complete Riftbound card list — the foundation every other feature builds on.

  • Search & domain filters: instant name search plus one-tap chips for the six domains and card type.
  • Set navigation: a “Your sets” view with per-set completion, drilling into any set with owned / on-loan / collected columns.
  • Progress at a glance: each set carries a completion bar so a collector can see how close a set is to done.
Card detail

The card data, exactly as Riot publishes it.

Every card opens to its full reference: art, full name and subtitle, domains, type, stats, and every printing that shares its name.

  • Variant awareness: base, showcase, and secret printings are modeled as distinct cards linked to their canonical name.
  • Legality & errata: format-scoped bans with effective dates and dated, cumulative errata are surfaced right on the card.
  • Your copies inline: owned and wanted counts per finish, aggregated by full name for playset math.
Deck builder

Build to the rules, and see what’s missing.

A full construction tool for Riftbound’s deck structure: one Legend, a Chosen Champion, a Main Deck, a Rune Deck, three Battlefields, and an optional Sideboard.

  • Zone-aware editing: each card goes to its correct zone, and you pick the exact printing you want for its art.
  • Full legality checklist: zone sizes, copy limits by full name, the Signature limit, Domain Identity, the champion tag match, and current bans — each rule passes, fails, or waits on a prerequisite.
  • Deck vs. collection: every entry shows how many copies you have available versus how many the deck needs.
Decklists & sharing

Share a deck, or match its gaps to your friends.

Decks are private by default. Share one — live and revocable — with specific friends, or move a list in and out as plain text.

  • Import & export: read and write decklists in the community text format, with a preview-first import that lists any unresolved lines before creating.
  • Per-friend sharing & copy: a shared deck is read-only; a friend can snapshot it into their own decks.
  • “Who has it?”: a deck’s missing cards are matched against friends’ available copies, with a prefilled loan proposal ready to send.
Loans & friends

Lend cards without losing track of them.

Friends connect through single-use QR invite codes — no search, no strangers. From there, loans move through a clear, bilateral lifecycle.

  • Loan lifecycle: proposed → accepted → active → returned, with declined, cancelled, expired, and a lender-side force-close, plus item-by-item partial returns.
  • Action inbox: the app derives the single move each loan is waiting on and badges the Loans tab with the count.
  • Accurate counts: lending never changes ownership — lent copies stay owned but stop being available, and never count in the borrower’s collection.
Wishlist

Track what you’re still hunting for.

Every collection entry carries a wanted quantity alongside the owned one; a dedicated wishlist gathers everything you’re after — useful at trade tables and when friends check what to lend.

  • Per card & finish: wanted counts are tracked exactly like owned copies, normal or foil.
  • Matching in both directions: see which friends have your wished cards available, and which of your cards sit on a friend’s wishlist.

How RiftKeep uses the Riot Games API

One thing, one path: the Riftbound card catalog.

RiftKeep needs exactly one input from Riot — the Riftbound card catalog (names, sets, collector numbers, domains, types, rarities, and card-art references). No player data, no match data, and no account data are requested from any Riot API.

Architecture · server-side sync only

The mobile app never calls the Riot API directly.

The catalog is synced server-side into the app’s own read-only database (Supabase / PostgreSQL); clients read only from that database.

AspectDesign
API consumerA single server-side sync job (Supabase Edge Function). Client apps have no Riot API access path.
API key storageServer-side environment secret only. The key is never embedded in the app binary, shipped to devices, or committed to source control.
Endpoint usageThe Riftbound Content API bulk catalog endpoint, fetched in full per sync run.
Request volumeThe catalog is near-static, so syncs are scheduled (about one run per day) plus occasional manual runs on set releases or errata updates — a handful of requests per day, independent of user count.
CachingThe synced catalog is versioned (version / lastUpdated); clients cache it for 24 hours and refetch only when the version changes.

Data handling

Catalog data is reference-only

  • ✓Stored as a read-only reference table; user data references it by foreign key but never modifies it.
  • ✓Displayed for reference inside the app only — never re-exposed through any public API, feed, or export.
  • ✓Ban-list and errata data are kept in sync with Riot’s official Rules Hub publications.

Not requested

What RiftKeep never touches

  • ✕Player accounts, summoner data, or Riot account links.
  • ✕Match, ranked, or any live-game telemetry.
  • ✕Any endpoint beyond the Riftbound card catalog.

User data & privacy

Accounts exist to hold your own cards — nothing more.

  • Accounts persist a user’s own collection, decks, and loans across devices (email-based authentication via Supabase Auth).
  • The only public profile field is a display name. No real names, avatars, locations, or Riot account links are collected.
  • Row-level security means users read and write only their own data, plus confirmed friends’ collections (read-only) and loans they are a party to.
  • No analytics or tracking SDKs, no advertising identifiers, no data sharing with third parties.

Technical summary

Built on Expo and Supabase.

PlatformiOS and Android, built with React Native / Expo (Expo Router, React Query, NativeWind).
